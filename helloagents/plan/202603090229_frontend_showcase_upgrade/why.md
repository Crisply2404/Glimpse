# 变更提案: 把 Glimpse 升级成更像前端项目的展示型 Demo

## 需求背景
现在这个项目已经有“流式返回 + 候选池 + 时间线 + 扭蛋结果”的雏形，但它更像“把 Agent 结果展示出来”，还不够像“前端同学主导设计的复杂交互系统”。

从简历角度看，当前最吃亏的点有两个：

1. **前端亮点偏静态展示。** 现在用户只能看一遍默认播放流程，不能暂停、拖动、回看、对比两次检索结果，也不能很直观地感受到“我刚加的线索到底改变了什么”。
2. **缺少真正有说服力的前端指标。** 现在能说的多是系统参数（比如 TopK、搜索轮数），但面试官真正关心的是：你有没有做复杂状态编排、有没有做性能优化、有没有把交互体验和可访问性做扎实。

这次规划的目标，不是去改“简历措辞”，而是先把项目本身补成一个更有前端含量的作品。等这些能力落地以后，简历里的前端亮点才会更硬。

## 产品分析

### 目标用户与场景
- **用户群体:** 想靠模糊线索找回目标的人；以及会从项目里评估前端能力的面试官/评审者
- **使用场景:** 用户不断补线索、回看过程、比较前后两次结果差异；面试官快速看到复杂交互、状态设计和性能意识
- **核心痛点:** 现有 Demo 更像“一次性播放结果”，还不是“可以操作、可以回放、可以比较、可以量化”的前端产品

### 价值主张与成功指标
- **价值主张:** 把“Agent 检索结果展示页”升级成“可回放、可比较、可量化”的前端交互系统
- **成功指标（本次作为目标值，不是当前已有数据）:**
  - 用户能暂停、续播、拖动进度、切换倍速，并回看任一步骤
  - 用户补充线索后，能直接看到前后两次结果的排名变化和淘汰变化
  - 事件量提升到 200~300 条时，页面仍保持流畅交互，没有明显掉帧或卡死
  - 键盘可完成核心操作，减少动效模式下仍能完整使用主要流程
  - 开发态能直接看到首条流式结果时间、总回放时长、当前候选量等前端指标，为后续简历提供真实数据

### 人文关怀
- 对容易晕动或不喜欢强动画的用户，提供减弱动效/手动播放能力
- 对只用键盘的用户，保证核心流程可操作，不把重要信息只藏在鼠标 hover 里
- 对结果不确定的场景，优先展示“变化依据”和“证据来源”，避免只给结论不给解释

## 变更内容
1. 增加“推理播放器”能力：支持暂停/继续、单步前进后退、0.5x/1x/2x 倍速、进度拖动和跳步回看。
2. 增加“前后两次检索对比”能力：展示排名升降、候选新增/淘汰、线索变化带来的影响。
3. 重构前端状态模型：把“流式接收状态”和“播放展示状态”拆开，减少当前单组件里过重的状态耦合。
4. 加入前端性能观测与体验优化：记录首条流式结果时间、回放总时长、候选量等指标，并针对长时间线/大候选池做渲染优化。
5. 补齐可访问性与细节体验：键盘操作、`prefers-reduced-motion`、主题持久化、焦点态与错误态优化。

## 核心场景

### 需求: 可回放推理播放器
**模块:** 前端（可视化）
用户不应该只能被动看一遍默认播放，而应该能像看录像一样控制推理过程。

#### 场景: 用户想暂停回看某一步
- 用户可以暂停、继续、切换倍速、拖动到某一步
- 当前高亮步骤、候选池状态、结果卡片要跟着播放进度同步变化
- 即使流式结果还没完全结束，用户也能回看已经到达的部分

### 需求: 检索结果前后对比
**模块:** 前端（可视化）
用户补了一条线索后，最想知道的不是“又返回了一组结果”，而是“和刚才比到底变了什么”。

#### 场景: 用户加了一条“不是像素风”的排除线索
- 页面能展示 Top 结果的升降名次、新增候选、被淘汰候选
- 结果区能看出哪些理由是新出现的，哪些证据导致了排序变化
- 用户不需要自己来回截图对比

### 需求: 长时间线也要保持顺滑
**模块:** 前端（可视化）
如果事件变多、候选池变大，页面仍然要稳，不能因为前端展示拖垮体验。

#### 场景: 一次请求产生 200~300 条事件
- 时间线滚动和候选池更新仍然顺滑
- 播放控制不会因为频繁重渲染而失灵
- 前端能输出可读的性能指标，帮助后续做真实优化和简历量化

### 需求: 核心交互对键盘和减弱动效友好
**模块:** 前端（可视化）
一个更成熟的前端项目，不应该只照顾“鼠标 + 默认动画”用户。

#### 场景: 用户开启减少动态效果，或主要用键盘操作
- 核心播放控制和结果浏览可以用键盘完成
- 开启减少动态效果后，自动播放和强动画可以降级，但核心信息不丢
- 焦点态、提示文案、错误提示保持清楚，不靠动画传达唯一信息

## 风险评估
- **风险:** 播放器、对比、性能观测同时上，会把 `GlimpseApp` 继续堆大
  - **缓解:** 先拆状态模型和派生选择器，再接 UI 组件，避免把所有逻辑继续堆在一个文件
- **风险:** 对比功能如果完全依赖后端，会把需求扩大成接口改造
  - **缓解:** 第一版先做前端本地快照对比，不新增后端接口
- **风险:** 性能面板如果默认对所有用户可见，会让主界面变杂
  - **缓解:** 默认仅在实验模式/开发模式显示，正式主流程保持干净
- **风险:** 新交互很多，容易让首次使用的用户懵
  - **缓解:** 播放控制默认保持“开箱即用”，高级能力放在二级区域，不干扰首次使用
