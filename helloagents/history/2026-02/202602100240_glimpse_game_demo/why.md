# 变更提案: Glimpse 游戏找回 Demo（Tavily + 可视化推理）

## 需求背景
用户常常只记得“零散印象”（年代、平台、画风、某个场景），但不知道游戏名字。直接问搜索/问答工具通常只给结果，不给“过程”。我们要做一个网页 Demo：把找的过程可视化，让用户能看见候选是怎么被筛选出来的。

## 产品分析

### 目标用户与场景
- **用户群体:** 想找回童年/记忆中的某款游戏的人；或者只记得一点点线索的玩家
- **使用场景:** 输入一段模糊描述 → 看见候选逐步收敛 → 继续补充线索 → 最后锁定游戏
- **核心痛点:** 不知道怎么搜；搜出来太杂；不知道为什么某个候选会被推荐

### 价值主张与成功指标
- **价值主张:** “不仅给答案，还把找的过程演给你看”
- **成功指标（Demo 阶段）:**
  - 本地一键运行
  - 输入描述后能得到 Top 结果 + 证据链接
  - 能看到至少 3 道“筛选关卡”的过程动画
  - 用户新增/修改线索后，排名能明显变化

### 人文关怀
- 不引导盗版/非法资源
- 不收集用户隐私信息（Demo 阶段不做账号）
- 解释尽量用大白话，不让用户被“专业术语”劝退

## 变更内容
1. 做一个 Next.js 网页 Demo（前端 + 后端 API）
2. 接入 Tavily 做全网搜索，生成候选游戏清单
3. 用大模型把“证据 + 打分理由”整理成可读内容，并产出“事件流”给前端做动画
4. 前端按事件流可视化展示：候选图标逐关筛掉 → 扭蛋机掉出 Top 结果

## 影响范围
- **模块:**
  - 前端（可视化）
  - 后端（API）
  - 推理流水线
  - 外部服务封装
- **文件:** 新增为主（新项目）
- **API:** 新增 `/api/recall`
- **数据:** 暂不落库（Demo 阶段以“请求内数据 + 缓存”完成）

## 核心场景

### 需求: 输入模糊描述也能得到候选
**模块:** 推理流水线
用户输入一段模糊描述，系统能从全网拉回候选并给出排名。

#### 场景: 第一次输入
用户只输入一句话，比如“像素风、俯视角、打怪升级、可能在 PC 上玩过”。
- 预期结果: 返回 Top 候选 + 证据链接 + 可解释的分数

### 需求: 用户能不断补充线索并看到变化
**模块:** 前端（可视化）

#### 场景: 边看边改
用户新增线索“不是回合制”“大概 2010-2015 年”，系统重新计算并展示新的筛选过程。
- 预期结果: 排名与筛选过程明显变化，用户能理解变化原因

### 需求: 过程可视化（筛选关卡 + 扭蛋机）
**模块:** 前端（可视化）

#### 场景: 回放筛选过程
系统把“候选从多到少”的过程拆成几关，前端能播放动画。
- 预期结果: 用户能看懂每一关筛掉了谁，以及为什么

## 风险评估
- **风险:** 全网搜索噪音大/同名多，候选会乱；大模型可能胡说
- **缓解:** 设计“证据强制”规则（没证据就不给高分）；做去重与多轮搜索；重要信息尽量引用来源链接
