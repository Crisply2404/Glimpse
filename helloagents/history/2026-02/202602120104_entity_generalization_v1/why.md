# 变更提案: 去“找游戏”痕迹 + 候选只掉落“本体”

## 需求背景
现在的 Demo 偏“找游戏”。当用户记忆很模糊时，我们会先全网搜网页，再从网页标题/摘要里提候选。问题是：**网页里有很多“求助帖/问句/盘点文章”**，它们的标题会被误当成候选，导致 Top5 里出现“网页/帖子”，而不是“你要找的那个东西本身”。

你希望把它升级为更通用的应用：用户对某个【东西/目标】有模糊印象，我们要把候选收敛到“目标本体”（例如：某个软件/网站/电影/书/游戏/产品等），并且尽量去掉界面上“游戏味”。

## 产品分析

### 目标用户与场景
- **用户群体:** 记忆模糊但愿意补线索的人（“我记得大概长这样/怎么用/什么时候见过”）
- **使用场景:** 想找回一个名字想不起来的东西（可能是游戏，也可能是软件、网站、电影等）
- **核心痛点:** 现有方案要么直接给答案（看不到推理过程），要么候选混进“网页标题/帖子问题”（不够像“找到了东西”）

### 价值主张与成功指标
- **价值主张:** 让用户看到“我们怎么一步步缩小范围”，并最终掉落“最像的候选”
- **成功指标（建议先用小目标）:**
  - Top5 候选中，**不再出现明显的“问句/求助帖标题/盘点标题”**（候选必须更像“一个具体名字”，而不是“文章标题”）
  - 首页文案/按钮/提示中，尽量不再出现“游戏”字样（除非用户输入里就是在找游戏）
  - UI 文案里不出现“事物”这个词（用更自然的说法，比如“那个/东西/目标/名字”）
  - 用户能在同一个区域里同时输入“整体描述 + 额外线索”，不用被“Clue Management”这种词搞懵

## 变更内容
1. **严格候选规则（通用版）:** 候选必须是“目标名字”，并且要能关联到至少 1 个“更像目标本体页”的证据链接；否则不允许进入 Top 排名。
2. **去游戏特化:** 后端提示词/默认搜索词不再强制“game”信号；前端文案尽量用中性表达（例如直接沿用现在那句：“输入模糊印象，我们会从全网找线索，把筛选过程演出来，最后用扭蛋掉出最可能的 Top 结果。”），不刻意强调“帮你找回什么”。
3. **线索输入更直观:** 去掉“Clue Management”这种概念，改成“高级条件（可不填）”的字段式输入（像高级搜索）：例如“一定包含 / 一定不包含 / 记得的时间或平台（随便写）”。
4. **搜索源对比:** 增加 Brave Search 作为可选搜索源，用同一套评估用例对比 Tavily vs Brave 的命中率与噪音。
5. **UI 对照测试（可选）:** 不把“选搜索源”塞给普通用户；提供一个“实验/对照”入口，让你能在两个页面里分别跑 Tavily/Brave（开两个浏览器标签页并行对比）。

## 核心场景

### 需求: 候选必须是“目标本体”
**模块:** 推理流水线
我们可以用网页当“证据来源”，但 **Top 候选必须是“目标名字”**，不能是“网页标题/帖子问句/盘点”。

#### 场景: 搜到很多求助帖/问句
- 候选池里可以留“线索页”，但 Top 排名只保留有“本体页证据”的候选
- 如果严格过滤后 Top 不够 5 个，宁可少给并提示用户补线索，也不要拿网页标题凑数

### 需求: 去“游戏特化”
**模块:** 前端（可视化） + 推理流水线
页面文案、后端提示词不要默认把目标当成游戏。

#### 场景: 用户找的是软件/网站/电影
- 搜索词生成不强塞“game/Steam”等词
- 解释型打分用词改为“目标/候选/证据”，而不是“游戏/关卡/玩法”
- 首页主文案保持中性（不说“找回XX”，只说“全网找线索→演示筛选→掉出 Top”）

### 需求: 线索输入更直观
**模块:** 前端（可视化）
用户应该在“初始描述”附近就能补线索，不需要理解“Clue Management”是什么；更像“高级搜索”的字段就行。

#### 场景: 用户想补充“不是X/大概年份/平台”
- UI 把它叫做“高级条件（可选）”，并且默认不展开（不懂也不影响使用）
- 字段式：用“一定包含 / 一定不包含 / 我记得在…（随便写）”这种更直白的输入
- 让用户一眼知道：不填也能搜；填了会影响排序（因为会帮你筛得更严）

### 需求: 搜索源对照测试（不打扰普通用户）
**模块:** 前端（可视化） + 外部服务
普通用户只管“输入描述 → 看筛选过程 → 掉落结果”，不需要理解“搜索源”是什么。
但我们自己做效果对比时，需要能方便地切换/对照 Tavily 和 Brave。

#### 场景: 需要对比 Tavily vs Brave
- 主页面不出现“选搜索源”入口（避免让人困惑）
- 提供两个实验页：`/lab/tavily` 与 `/lab/brave`
- 两个页面 UI 一样、输入一样，只是背后用的搜索源不同（可以开两个浏览器标签页并行跑）

## 风险评估
- **风险:** 严格过滤后候选数量变少 → 用户觉得“怎么只掉 2~3 个”
  - **缓解:** 明确提示“严格模式下宁可少给也不胡凑”，并引导补充线索；后续可加“我愿意更慢/更贵，给我更多候选”按钮（但默认不自动补搜）
- **风险:** AI 自动识别“目标类型”可能误判
  - **缓解:** 做降级策略：识别不出来就走“通用模式”，允许更多类型的本体页来源
