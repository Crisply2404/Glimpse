# 技术设计: 去“找游戏”痕迹 + 候选只掉落“本体”

## 技术方案

### 核心技术
- Next.js（前端 + API）
- Tavily（搜索，当前默认）
- OpenAI（可选：生成搜索词、提炼候选、解释型打分）

### 实现要点（大白话）
1. **先猜“你在找什么类型的东西”**（让 AI 自动识别，不做下拉）
   - 例如：更像软件/网站/电影/书/游戏/产品/歌曲/不确定
   - 猜不出来就当“不确定”，走通用逻辑
2. **搜索只负责捞网页，候选必须像“目标本体”**
   - 网页可以是证据来源，但不能把“问句/求助帖标题/盘点标题”当候选
   - 候选进入 Top 之前，必须能关联到至少 1 个“本体页证据”
3. **线索输入改成“高级搜索字段”**
   - 不再叫 Clue Management
   - UI 用“高级条件（可选）”折叠区：一定包含 / 一定不包含 / 我记得在…（时间/平台随便写）
   - 后端仍然把这些字段转换成“线索（是/不是）”，用于打分与解释（用户不用懂内部结构）
3. **“本体页证据”是通用概念，不等于“游戏页”**
   - 游戏：Steam / 主机商店 / Wikipedia / 官方站
   - 软件/开源项目：官网 / GitHub 仓库 / npm / PyPI
   - App：App Store / Google Play / Microsoft Store
   - 电影/书：Wikipedia / IMDb / Goodreads（先从 Wikipedia 开始，避免引入太多外部依赖）
4. **去游戏特化**
   - 后端提示词不再默认“游戏”，改为更通用的说法（例如“根据模糊印象找候选”）
   - 用户侧文案保持中性：不刻意说“帮你找回什么”，而是用“全网找线索→演示筛选→掉出 Top”这种描述
   - 搜索词生成不再强制包含 game/Steam 等词

## 架构设计（流程）
1. 输入：用户的模糊描述 + 线索（包含“是/不是”）
2. 推理流水线：
   - (A) 识别目标类型（可选：OpenAI）
   - (B) 生成搜索词（根据类型动态生成“信号词”）
   - (C) 多轮搜索合并去重
   - (D) 提炼候选（要求：候选名像目标名，证据来自命中的 url）
   - (E) **严格候选校验：必须有本体页证据**（否则降级为“线索页”，不进 Top）
   - (F) 打分排序（解释型打分继续沿用“有引用才加减分”的约束）

## 架构决策 ADR

### ADR-001: 严格候选=目标本体（不再接受“网页标题候选”）
**上下文:** 全网搜索里大量结果是求助帖/盘点文章，标题经常像“问句”，会污染 Top5。  
**决策:** Top 候选必须满足“像目标名 + 有本体页证据”，否则不进入 Top。  
**理由:** 用户要的是“找到东西”，而不是“找到一堆网页”。  
**替代方案:** 继续允许网页标题当候选 → 拒绝原因: Top5 质量不可控、用户信任感下降。  
**影响:** 结果更干净，但可能候选数量变少，需要更好的提示与“扩池按钮”（后续再做）。

### ADR-002: 目标类型用 AI 自动识别（不做强制下拉）
**上下文:** 未来要找的不止游戏，不希望 UI 逼用户先选类别。  
**决策:** 后端先做一次“类型猜测”，猜不准则走通用逻辑。  
**理由:** 保持 UI 简洁，同时让搜索词更贴近目标。  
**替代方案:** UI 下拉让用户选择类型 → 拒绝原因: 增加输入负担，用户可能也不知道该选啥。  
**影响:** 需要降级策略，避免误判导致搜索偏航。

## API设计
不新增外部 API；对普通用户仍然用默认搜索源。

如果要做“对照测试”，建议加一个**仅实验页使用**的请求参数（比如 `options.provider=tavily|brave`），让同一个部署里可以开两个页面分别跑两家（不用把“选搜索源”塞给所有用户）。

## UI 对照测试（Brave vs Tavily）
- **推荐做法（简单、实用）:** 做两个实验页：`/lab/tavily` 和 `/lab/brave`。
  - 你可以开两个浏览器标签页并行跑同一段描述，体验/结果一眼对比
  - 主页面不展示“选搜索源”，避免普通用户困惑
- **后续可选（工作量更大）:** 同一页面左右并排对照（更像评测台）

## 搜索源选择（为什么先 Tavily + Brave）
- **Tavily：** 给的摘要通常更长，更适合我们做“证据原文片段”（也就是用户点开能看到那句原话）
- **Brave：** 更像“传统搜索引擎”的结果，覆盖面/多样性常常更好，适合做对照组
- **其它候选（不是不考虑，只是先不一起上，避免变成大工程）：**
  - Exa：语义搜索更强（模糊描述可能更容易命中），但要评估成本与返回格式
  - Google CSE：更“官方/稳”，但配置麻烦、配额限制也更明显
  - Serper/SerpAPI：更像 Google 结果，但成本/稳定性/规则风险不确定
  - 自建 SearXNG：可控但需要运维

## 安全与性能
- **安全:**
  - 不把任何 API key 传到前端
  - 外部请求都带超时；失败可降级（继续返回事件流 + 提示）
- **性能:**
  - 继续使用短期缓存避免重复扣费
  - “严格候选校验”优先用 URL 规则（不额外抓网页正文），避免变慢

## 测试与部署
- **测试:**
  - 新增用例：当命中里混入“问句/求助帖标题”时，这些不应该出现在 Top 候选中
  - 新增用例：当候选没有任何“本体页证据”时，应该被过滤掉（或降级为线索，不进 Top）
  - 搜索源对比：用同一批样例跑 Tavily vs Brave，记录 Top1/Top5 命中率、噪音占比
- **部署:**
  - 只涉及环境变量新增（Brave key），无需迁移
