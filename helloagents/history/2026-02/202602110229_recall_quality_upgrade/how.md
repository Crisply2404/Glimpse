# 技术设计: 找回质量升级（候选池下限 + 证据约束推理）

- OpenAI（提炼候选 + 解释型打分，现有）
- NDJSON 流式输出：`POST /api/recall/stream`（现有）

## 实现要点（大白话版）

### 1) 证据约束推理（通用规则，不做“词语补丁”）
目标：不管是平台、玩法、画风还是任何结论——**只要你说“因为所以”，就必须能在证据片段里找到对应文字**。

实现思路：
1. **让大模型输出“证据原话”**：
   - 在解释型打分的 JSON 里新增字段，比如 `evidenceQuote`（一句短短的原话/片段）
   - 要求 `evidenceQuote` 必须是“我们传进去的 evidence.snippet 的子串”（也就是能被程序验证）
2. **后端做硬校验**：
   - 如果 `evidenceQuote` 在对应 snippet 里找不到 → 这条理由直接丢弃或降权，并在 reason 里写“证据不足”
   - 这样就能从机制上逼近“别瞎猜”，而不是针对某些词特判
3. **前端展示更可信**：
   - 在结果卡片里，理由旁边可以显示一小段引用（可选），并且链接到对应证据页

收益：
- 不会出现“因为是 Steam 所以就是 PC-only”这种跳步推理（除非 snippet 里真的写了 PC-only / Windows / macOS 等）
- 任何结论都能追溯到“证据原话”

---

### 2) 候选池下限（让筛选真的发生）
目标：候选池太小，筛选就不成立。我们要保证“通常能看到一堆候选被淘汰，只剩 Top5”。

实现思路：
1. **定义目标候选数**：例如 `desiredMin = max(12, topK * 3)`，再受 `maxCandidates` 限制
2. **第一次提炼后检查数量**：
   - 如果候选数量 `< desiredMin`：
     - 触发一次“自动补搜”：用更宽的关键词（玩法机制 + 平台关键词）再搜一轮
     - 再提炼一次候选，然后合并去重
3. **还不够就兜底补齐**：
   - 从搜索命中的网页标题里按规则提取“像游戏名的短标题”，当成低置信候选（后续会被筛掉）
   - UI 可以不专门标注“低置信”，也能通过分数/淘汰体现（如果要更透明，可以加一个小标签）

事件流（可视化）上需要补几条更清楚的事件：
- “候选偏少 → 正在自动补搜拓宽候选池…”
- “补搜完成 → 候选池扩充到 N 个”

---

### 3) （可选）搜索源可替换：别急着换，先把对比做出来
你怀疑 Tavily 可能导致“漏搜/不准”，这个判断完全合理，但不建议现在立刻拍脑袋换掉。

推荐做法：
1. 先把搜索提供方抽象成接口（比如 `searchProvider.search(query)`）
2. Tavily 作为默认实现
3. 再加第二个实现（例如 Brave / Serper / Bing 等）
4. 用同一批输入做 A/B 对比（候选数、Top 命中、证据质量、耗时、成本），再决定要不要换

这样做能避免“换完还是不准”——因为根因可能是“提炼/打分环节”而不是搜索源。

## 架构决策 ADR

### ADR-004: 解释型打分必须输出可验证的证据引用（evidenceQuote）
**上下文:** 当前模型会写出看似合理但证据不足的结论，用户不信任。  
**决策:** 要求模型输出 `evidenceQuote`，并由后端校验它必须来自 evidence.snippet。  
**理由:** 这是“通用机制”，不依赖特定关键词；能系统性抑制瞎猜。  
**替代方案:** 针对“平台/年份/画风”等词做特判 → 拒绝原因: 亡羊补牢，越补越乱。  
**影响:** 返回结构会更严格；少量理由会被丢弃/降权，但整体可信度提升。

## 安全与性能
- Key 仍只在后端（`.env.local`），前端不接触
- 自动补搜要有上限，避免无限扩散
- 图片抓取只解析 meta（og:image 等），并且有超时 + 缓存，避免拖慢主流程
